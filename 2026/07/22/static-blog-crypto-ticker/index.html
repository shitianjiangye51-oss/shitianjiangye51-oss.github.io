给静态博客接实时加密行情：一次撞上两堵墙后撤退的复盘

本文最后更新于 2026-07-22T09:59:03+08:00

想给纯静态的博客加一个「会说人话」的实时加密行情页，结果撞上了两堵墙——一堵是国内的网络长城（GFW），一堵是浏览器的同源策略（CORS）。这是一次从兴致勃勃到坦然撤退的完整复盘。

一句话结论

对一个纯静态前端来说，”免费 + 无密钥 + 允许浏览器直连 + 国内可访问”的加密行情 API，客观上不存在。 不是没找到，是这四个条件的交集是空集。最后我把整个行情页下掉了——但过程里踩的坑值得记下来。

起点：一个想「说人话」的行情页

最初的想法很简单：在博客上放一个实时加密行情页，别做成冷冰冰的报价表，要好玩、会说人话。做出来的版本包含四个板块：

  • 😱 恐惧与贪婪指数：半圆 SVG 仪表盘 + 指针，配一句应景吐槽；
  • 🌍 全球资产市值排行榜：把 BTC、ETH 和黄金、白银、英伟达、苹果等放一起排队，看比特币现在全球第几；
  • 🕹️ 假如当时上车：输入「一年前花 1000 刀买 BTC」，算出现在赚了几倍还是亏麻了；
  • 💰 主流币实时行情：10 张币卡，带走势小图和拟人化心情文案（「🚀 起飞了，追还是不追？」）。

技术上是一个自包含的 Markdown 页：内联 <style> + 内联 <script>，浏览器直接 fetch 第三方 API。关键约束就藏在这里——静态站没有后端，所有数据必须由浏览器直接去拉。而浏览器直连第三方接口，必须过 CORS 这一关。

第一堵墙：GFW —— CoinGecko 在国内加载不出来

第一版用的是 CoinGecko，海外开发常用、数据全、有正确 CORS。但用户在国内一打开：转圈半天，价格出不来。原因很直白——CoinGecko 在国内被墙，请求要么超时要么被重置。

于是目标变成：换一个国内能直连的数据源。听起来只是换个 URL，实际是噩梦的开始。

第二堵墙：CORS —— 有数据 ≠ 能用

我逐个实测了主流交易所的接口，判定标准有两条：①国内网络能不能连通；②响应头有没有正确的 Access-Control-Allow-Origin。第二条最容易被忽略，却是静态站的生死线。

数据源 国内连通 CORS 头 能否前端直连 结论
OKX 时好时坏 ✅ 正确回显 origin 连得上就能用 勉强可用，见下文
Gate.io ✅ 能连 ❌ 缺 Access-Control-Allow-Origin 被浏览器拦 有数据也白搭
Binance ❌ 被墙 — 连不上 弃用
新浪 / 腾讯财经 ✅ 能连 ❌ 无 CORS 头 被浏览器拦 有价格也用不了
CoinGecko ❌ 被墙 ✅ 正确 国内连不上 只能作海外回退

Gate.io 这一栏最能说明问题：接口能返回完整的 BTC 价格，curl 拿得一清二楚，但浏览器里就是拿不到——因为它只给了 Access-Control-Allow-Methods、Access-Control-Allow-Headers，唯独没给最关键的 Access-Control-Allow-Origin，浏览器在跨域时直接把响应拦在门外。国内财经巨头新浪、腾讯的行情接口同理：数据就在那里，但它们从没打算让别的网站的前端直接调，自然不会配 CORS。

💡 教训一：curl 能拿到，不代表网页能拿到。 命令行没有同源策略，浏览器有。评估一个”能不能前端直连”的接口，必须看它的响应头，而不是看数据。

挣扎期：OKX + 自动回退链

OKX（欧易）成了唯一勉强能打的选手——它有正确的 CORS。于是我做了一条数据源自动回退链：

按顺序尝试，谁先成功就锁定谁：国内用户走 OKX，海外/翻墙环境走 CoinGecko。同时顺手解决了用户反馈的另外两个体验问题：

  • 「刷新太慢」：给每个请求加了 AbortController 超时（8~9 秒）。某个源被墙时快速失败、立刻回退，而不是干等浏览器自己超时（默认可能几十秒）；
  • 「感觉跳动」：以前每次刷新都整块重建卡片 DOM，导致图片重载、布局重排，看着就”抖”。改成卡片只构建一次，之后只原地更新价格 / 涨跌 / 心情的文字，价格变化用数字滚动 + 涨绿跌红闪一下。顺带把依赖图片 CDN 的币种 logo 换成了纯 CSS 彩色字母徽章（图片 CDN 也可能被墙）。

这一版在能连上时体验相当不错。但问题的根子没解决——

撞墙时刻：OKX 也不是「国内 API」

真相是：OKX 早已退出中国大陆，www.okx.com 在大陆很多网络环境下同样连不上。 它有 CORS，不代表它在国内稳定可达。所谓”国内优先走 OKX”，在相当比例的用户那里依然是转圈。

至此，四个条件的交集彻底被证明是空的：

  • 国内能连的（新浪/腾讯）→ 没 CORS，前端用不了；
  • 有 CORS 的（CoinGecko/OKX）→ 要么被墙，要么在国内不稳定；
  • Binance → 直接被墙。

💡 教训二：“有 CORS” 和 “国内可访问” 这两个属性，在免费加密行情 API 里几乎是互斥的。 合规能进国内的数据源不会开放跨域直连；开放跨域的国际接口进不了国内。

坦然撤退

既然客观上不存在满足条件的接口，继续在前端硬凑只是在制造”薛定谔的行情页”——能不能看全看运气。和用户确认后，我把整个「加密行情」页连同导航入口一起下掉了。

这不是失败，是把工程判断力用在了正确的地方：当约束证明目标不可达时，及时撤退比继续堆代码更有价值。

真正的解法（如果一定要做）

问题的本质不是”找不到数据”，而是”静态前端不能安全地跨域直连”。所以正解是加一层自己的后端中转：

方案 做法 代价
云函数 / Serverless 用 Cloudflare Workers、Vercel、腾讯云函数等写个代理，服务端去拉数据（服务端没有 CORS 限制），再给自己的响应加上 CORS 头返回 需要部署，可能有额度/冷启动
自建小服务 一台能同时访问数据源和国内的服务器定时抓取、缓存、对外提供带 CORS 的接口 需要服务器和运维
静态快照 构建时抓一次价格写进页面，牺牲实时性 数据不新鲜

一旦有了带 CORS 的中转层，前端就能稳定拿到数据，”国内能不能连数据源”这个问题也转移到了服务端——服务端选一个自己能访问的源即可。这才是给静态站接第三方数据的正确姿势。

这次折腾最大的收获，不是做出了一个行情页，而是彻底搞懂了：静态博客接实时数据，真正的门槛是 CORS，不只是那堵网络长城。 下次再想给静态站加动态数据，我会先问一句——“这个接口，浏览器拿得到吗？”
